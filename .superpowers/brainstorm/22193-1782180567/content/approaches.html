设计决策汇总

调用方式 Python 库 + CLI 命令行
评分卡对接 抽象接口/协议，pkl 路径为便捷方式之一
定制空间 核心结构固定，列名/标签/分区名可配置
变量元数据 CSV/YAML 变量清单文件
设计原则 默认代替配置

三种架构方案

点击选择一个方案

A

单体 ReportGenerator 类

一个类 + ExcelWriter 工具，所有 sheet 生成逻辑在一个文件中。参考 scorecard_jsb.py 的风格，简单直接。

Pros

  • 文件少，易于理解
  • 适合初期快速迭代
  • 与现有代码风格一致

Cons

  • 文件会很大（3 个 sheet 逻辑 + Excel 格式化）
  • 难以单独测试某个 sheet
  • 后续扩展困难
B

按 Sheet 拆分模块

每个 sheet 独立为一个模块，ReportGenerator 作为编排层。接口层独立。

model_report/
├── __init__.py # 公开 API
├── cli.py # CLI 入口
├── interface.py # ScorecardProtocol 抽象接口
├── config.py # 默认配置 + 列名映射
├── metadata.py # 变量清单加载
├── generator.py # ReportGenerator 编排器
├── sheets/
│ ├── model_design.py # Sheet 1
│ ├── variable_analysis.py # Sheet 2
│ └── model_performance.py # Sheet 3
└── writer.py # Excel 写入 + 条件格式

Pros

  • 每个 sheet 独立可测
  • 模块职责清晰
  • 易于扩展新 sheet

Cons

  • 文件较多
C

管道/中间数据模型

在 B 的基础上，中间计算结果用 dataclass 明确定义。Sheet 生成器接收结构化中间数据而非原始 DataFrame。

model_report/
├── ... (与 B 相同)
├── models.py # 中间数据 dataclass
│ # PartitionStats, VariableStats,
│ # ScorecardDetail, ModelEffect,
│ # BinPerformance
└── sheets/ (同上)

Pros

  • 计算与展示完全解耦
  • 中间数据可独立复用（如输出 JSON）
  • 最佳可测试性

Cons

  • 层级多，初期开发量稍大
  • dataclass 定义需要维护
